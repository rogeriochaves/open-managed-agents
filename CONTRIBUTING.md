# Contributing to Open Managed Agents

Thanks for your interest in contributing. This project is a self-hosted, open-source clone of Anthropic Claude Managed Agents, with multi-LLM support and enterprise governance on top. The goal is to be a dependable, well-tested platform that enterprises can run internally with confidence.

## Repository layout

It's a pnpm + Turborepo monorepo.

```
packages/
  types/               Shared TypeScript types (zero deps)
  server/              Hono API server, SQLite + Postgres, engine, routes
  web/                 React 19 + Vite 6 + Tailwind v4 frontend
  cli/                 `oma` CLI, wraps the Anthropic SDK against a self-hosted base URL
  scenario-tests/      End-to-end tests via LangWatch Scenario (opt-in, needs real LLM keys)
helm/                  Helm chart (sqlite / embedded postgres / external postgres)
specs/                 Gherkin specs, one file per feature or bug
docs/screenshots/      Generated by scripts/capture-screenshots.mjs
scripts/               Screenshot capture + CLI smoke test
.github/workflows/     CI: test + sqlite-smoke + helm + docker-compose
```

## Development setup

```bash
pnpm install
cp .env.example .env    # add at least one LLM API key
pnpm dev                # runs server on :3001 and web on :5173
```

Admin login: `admin@localhost` / `admin`. Change via `OMA_DEFAULT_ADMIN_PASSWORD`.

## Tests

Run the full suite:

```bash
pnpm test        # server 102 + web 88 + cli 5 + scenario 2-skipped (197 total)
pnpm typecheck   # across every package
pnpm build       # across every package
```

Per-package:

```bash
pnpm --filter @open-managed-agents/server test
pnpm --filter @open-managed-agents/web test
pnpm --filter @open-managed-agents/cli test
```

End-to-end scenario tests against a real LLM (opt-in, costs money):

```bash
OMA_SCENARIO_ENABLED=1 pnpm --filter @open-managed-agents/scenario-tests test
```

These require `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` (the judge uses `gpt-5-mini`).

## How to add a new route

1. Define the Zod schemas in `packages/server/src/schemas/`.
2. Register the route with `createRoute` in `packages/server/src/routes/`.
3. Add an integration test in `packages/server/src/__tests__/` that uses `createApp({ skipProviderSeed: true })` + `app.request()` — no real HTTP server, just in-process.
4. If the route mutates state that should be audit-logged, call `auditLog(currentUserId(c), ...)` after the mutation.
5. Add a Gherkin spec in `specs/` describing the scenarios.

## How to fix a bug

1. Add a failing test in `packages/server/src/__tests__/` or `packages/web/src/__tests__/`.
2. Fix the code so the test passes.
3. Write a `specs/<bug-name>.feature` file explaining the root cause in the background / comments, and the scenarios that now prevent regression.
4. Reference the spec file in the commit message.

See `specs/audit-auto.feature` and `specs/cli-client.feature` for examples — both documented latent bugs found by tests.

## Code style

- TypeScript strict mode on everywhere.
- No comments unless the *why* is non-obvious (hidden constraint, known workaround, subtle invariant). Well-named identifiers do the *what*.
- Only add error handling / fallbacks at real system boundaries (user input, external APIs, DB).
- Prefer editing existing files over creating new ones.
- No emojis in source unless explicitly needed.

## Conventional Commits

Use [Conventional Commits](https://www.conventionalcommits.org/) prefixes:

- `feat(scope): …` new feature
- `fix(scope): …` bug fix (link the spec file)
- `test(scope): …` test-only change
- `docs: …` docs-only change
- `ci: …` CI/workflows
- `chore: …` tooling, deps, formatting
- `refactor(scope): …` no behavior change

Scope is usually the package name: `server`, `web`, `cli`, `types`, or `infra`.

## CI

Every PR must pass:

- `test` — typecheck + tests + build (all 5 packages)
- `sqlite-smoke` — boots the built production server with SQLite and drives the compiled `oma` CLI against it end-to-end
- `helm` — lints + renders the chart in sqlite / embedded-postgres / external-postgres modes and sanity-checks the kinds
- `docker-compose` — validates `docker-compose.yml` parses and declares the expected services

No PR is green until all four pass. See `.github/workflows/ci.yml`.
